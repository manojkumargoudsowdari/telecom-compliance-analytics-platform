# Phase 4-3: Raw-to-Silver Transformation Rulebook

## 1. Overview

This document defines the transformation behavior that converts Phase 3
FCC raw complaint records into the persisted Silver dataset
`silver_fcc_consumer_complaints`.

It builds on:

- `docs/phase4/phase4-1-silver-functional-spec.md`
- `docs/phase4/phase4-2-silver-schema-contract.md`

This rulebook defines transformation behavior only. It does not define
implementation design, orchestration, or code-level mechanics.

## 2. Transformation Scope

- Input:
  - FCC raw JSON page files landed under
    `data/raw/fcc/consumer_complaints/{run_id}/`
  - corresponding raw metadata files under `data/raw/fcc/_metadata/`
- Output:
  - persisted Silver dataset `silver_fcc_consumer_complaints`
- Grain target:
  - `one latest valid record per complaint_id`

The rulebook applies to the conversion of raw FCC record objects into
the approved Silver schema and to the deterministic selection of a
single final Silver row per `complaint_id`.

## 3. Transformation Principles

- Deterministic: the same raw inputs must yield the same Silver outputs
  when the same rule set is applied
- Traceable: every Silver record must remain attributable to a specific
  raw ingestion run
- Schema-aligned: every produced Silver row must conform to the Phase
  4-2 schema contract
- Minimal semantic reinterpretation: raw values should not be renamed or
  reinterpreted beyond the approved Silver contract
- No silent record loss: records excluded from final Silver must be
  explicitly treated as invalid or superseded by deduplication rules
- Documented null handling: empty, placeholder, and invalid values must
  follow explicit rules
- Reproducible across runs: rerunning the same transformation over the
  same raw inputs must not change the selected Silver record set
- Idempotent: repeated execution over identical raw inputs produces
  identical Silver outputs

## 5. Structural Cleanup Rules

- Trim leading and trailing whitespace from all string-source fields
  before null checks or type casting
- Treat whitespace-only strings as empty values
- Collapse empty strings to `NULL` for optional columns
- Apply only schema-approved column renames:
  - `id` -> `complaint_id`
  - `zip` -> `zip_code`
  - `type_of_call_or_messge` -> `type_of_call_or_message`
- Preserve source wording and source case for business text fields
  unless the cleanup is contract-safe and low-risk
- Apply uppercase normalization to `state`
- Do not introduce undocumented semantic renaming such as
  `date_created` -> `date_received` or `method` -> `submitted_via`
  during this phase

## 6. Type Casting Rules

### String Fields

- Convert source values to their trimmed string representation where
  applicable
- Keep business text as `STRING`
- Preserve identifiers such as `complaint_id` and `zip_code` as strings

### Date Fields

- `date_created` and `issue_date` must be interpreted as `DATE`
- Required date parse failure for `date_created` makes the record
  invalid for final Silver
- Optional date parse failure for `issue_date` results in `NULL`

### Timestamp Fields

- `ticket_created_utc` must be interpreted as `TIMESTAMP`
- Parse failures for `ticket_created_utc` result in `NULL` because the
  field is optional
- `silver_processed_at_utc` is assigned as a valid UTC timestamp by the
  Silver process

### `issue_time_raw`

- `issue_time_raw` remains `STRING`
- No attempt is made in this phase to parse or standardize it into a
  time datatype

### Invalid Cast Behavior

- Invalid casts on required fields make the record ineligible for final
  Silver
- Invalid casts on optional fields produce `NULL`
- Invalid records must not be silently discarded

## 7. Required Field Handling

The following fields are required for a valid final Silver record:

- `complaint_id`
- `date_created`
- `source_run_id`
- `silver_processed_at_utc`
- `source_system`

Expected handling behavior:

- if a required field is missing, empty, or invalid, the record must not
  enter final Silver
- if a required lineage field cannot be assigned, the condition must be
  treated as a transformation failure condition rather than a silent row
  omission
- records that fail the required-field contract must be routed to the
  reject or quarantine path defined in Phase 4-5

## 8. Deduplication Rule Hierarchy

Deduplication must be applied after records are mapped into the approved
Silver shape and after invalid records are excluded from final Silver
eligibility.

Rule hierarchy:

1. Partition candidate records by `complaint_id`
2. Prefer the record from the latest `source_run_id`
3. Determine latest `source_run_id` by chronological ordering of the
   UTC timestamp embedded in the Phase 3 run ID
4. If run IDs share the same timestamp base and include a collision
   suffix, order by the numeric suffix after the timestamp
5. If multiple candidate records remain within the same `source_run_id`,
   prefer the record with the latest non-null `ticket_created_utc`
6. If a tie remains, prefer the record with the latest non-null
   `date_created`
7. If a tie still remains, prefer the record from the later raw page
   file based on the deterministic page sequence established during
   Phase 3 ingestion (limit-offset pagination order)
8. If a tie still remains within the same raw page, prefer the later raw
   row position within that page
9. If an exact tie still remains, apply a final deterministic fallback
   using a stable hash derived from the canonical serialized raw record
   content

This hierarchy defines the expected rule behavior. Exact physical
implementation belongs to later Phase 4 execution work.

## 9. Null and Empty-Value Handling

- `NULL` and empty-string behavior must be deliberate and documented
- whitespace-only values are treated as empty
- empty optional string values become `NULL`
- required fields do not permit null-equivalent values
- optional fields remain nullable in Silver when the source does not
  provide a usable value
- source placeholder absence values such as `None` may be normalized to
  `NULL` when they are clearly functioning as missing-value markers
- normalization should preserve information where a source value could
  still carry business meaning

## 10. Invalid Record Handling

- records that cannot satisfy the required-field contract must not enter
  final Silver
- records with optional-field parsing issues may remain eligible for
  final Silver if the affected optional fields are normalized to `NULL`
- invalid records must be routed to reject or quarantine handling
  defined in Phase 4-5
- invalid records must not be silently dropped

## 11. Out of Scope

This rulebook does not define:

- quality thresholds or pass/fail gates
- reject table schema
- implementation logic
- execution or orchestration design
- Gold-layer modeling, KPIs, or aggregates

## 12. Open Questions / Review Notes

- The final deterministic fallback uses a stable hash derived from the
  canonical serialized raw record content. Phase 4-6 should confirm the
  exact execution-safe mechanism (e.g., canonical serialization method
  and hash function) without changing the rule intent.
- Future architecture review may decide whether `date_created` and
  `method` should gain approved business aliases, but that is not part
  of the current transformation rulebook.
